# 문장 담기 + PC 대시보드 — 설계 문서

## 배경 및 목표

Phase 3(비주얼 리디자인) 배포 후 사용자가 실제 사이트를 보고 나온 두 가지 요청.

**1. PC 화면이 비어 보인다.** 앱이 모바일 기준으로 만들어져서, 넓은 화면에서는 상단에 카드 몇 개만 뜨고 아래 70%가 빈 회색이다. 폰에서는 만족스럽다고 했으므로 **모바일 레이아웃은 건드리지 않는다.**

**2. 즉석 번역 기능이 없다.** 사용자의 실제 시나리오: "걷다가 갑자기 '집에 가고 싶은데 어떻게 할까'를 일본어로 뭐라고 하지?" 싶을 때 쓸 것이 없다.

### 번역 기능에 대한 결정 (중요)

임의의 한국어 문장을 실시간 번역하는 기능은 **만들지 않는다.** 이 앱은 서버가 없는 GitHub Pages 정적 사이트라 번역/LLM API 키를 숨길 곳이 없고, 코드에 넣으면 소스만 봐도 누구나 가져다 쓸 수 있다.

대신 **문장 담기(phrase capture)** 로 간다: 떠오른 한국어 문장을 앱에 저장해두고, 번역은 사용자가 외부 도구(파파고 등)에서 확인해 직접 채워 넣는다. 채워 넣은 문장은 곧바로 문어장·해석 연습·SRS 복습 큐에 합류한다.

사용자에게 세 가지 대안(본인 API 키 입력 / 서버리스 프록시 / 담기만)을 제시했고 담기 방식을 선택함. 학습 관점에서도 이쪽이 낫다는 판단: 번역기는 답을 즉시 주지만 그래서 기억에 남지 않고, 담아둔 문장이 며칠에 걸쳐 복습 큐로 다시 나타나는 쪽이 실제로 외워진다.

## 범위

- 기능 ①: 문장 담기 — 한국어 문장 저장, 일본어(+선택적 읽기) 채워넣기, 완성된 문장이 기존 학습 흐름에 합류, 내보내기/가져오기
- 기능 ②: 홈 화면을 대시보드로 — 넓은 화면에서 3열, 좁은 화면에서는 지금처럼 1열
- **제외**: 실시간 번역/API 연동, 서버·계정·기기 간 자동 동기화, 모바일 레이아웃 변경

## 기능 ①: 문장 담기

### 데이터 모델

`src/types.ts`에 추가:

```typescript
export interface CapturedPhrase {
  id: string;         // 'my-<타임스탬프>'
  korean: string;     // 담을 때 입력, 필수
  japanese: string;   // 채우기 전에는 ''
  reading: string;    // 선택 — 비어 있으면 작문 연습에는 안 나옴
  createdAt: string;  // 'YYYY-MM-DD'
}
```

- **완성 여부** = `japanese !== ''`. 별도 상태 필드를 두지 않는다(파생 가능한 값을 저장하면 어긋날 수 있음).
- `reading`은 **선택**. 사용자가 결정한 사항이며 이유는 아래 "읽기가 선택인 이유" 참고.
- `english`는 두지 않는다. 기존 `SentenceEntry`의 `english`는 내장 40문장을 위한 것이고, 담은 문장에까지 요구하면 입력 부담만 커진다.

저장 위치: `localStorage` 키 `captured-phrases`, `CapturedPhrase[]` 배열. SRS 상태는 기존 문장과 동일하게 `srs-store-sentences`를 쓴다(문장 id가 키이므로 그대로 섞여도 충돌 없음).

### 읽기(reading)가 선택인 이유

작문 연습은 가나 키보드로 입력받아 `reading`(가나 전용)과 정확 일치로 채점한다. 사용자가 `家に帰りたい`처럼 한자 표기만 넣으면 가나로는 절대 정답을 만들 수 없다. 따라서:

- `japanese`만 채운 문장 → 문어장 + **해석 연습**에 등장
- `japanese` + `reading` 둘 다 채운 문장 → 위 + **작문 연습**에도 등장

화면에 이 구분이 보여야 한다(예: 읽기가 없는 문장에 "해석 연습만" 배지). 그래야 사용자가 왜 어떤 문장은 작문에 안 나오는지 알 수 있다.

### 기존 학습 흐름에 합류시키는 방법

현재 5개 지점이 `SENTENCES.entries`를 직접 읽는다(`sentence-view.ts` 3곳, `interpret-view.ts`, `compose-view.ts`, `home-view.ts`). 각 지점에서 따로 합치면 빠뜨리기 쉬우므로, **합치는 로직을 한 곳에 둔다.**


### 화면

**홈의 문장 담기 카드** (대시보드 왼쪽 열): 한 줄 입력 + "담아두기" 버튼 + 채우기 대기 목록 미리보기(최대 3개) + "내 문장" 화면으로 가는 링크. 걷다가 떠오른 걸 즉시 던져넣는 용도이므로 입력은 한 칸뿐이다.

**내 문장 화면** (`#/phrases`, nav에 추가): 담은 문장 전체 목록. 각 항목마다
- 채우기 전: 한국어 문장 + 일본어 입력칸 + 읽기 입력칸(선택 표시) + 저장 버튼 + 삭제
- 채운 뒤: 완성된 문장 카드(기존 문장 카드와 같은 모양) + 수정/삭제, `reading`이 없으면 "해석 연습만" 배지

"채우기 대기"가 먼저, 완성된 것이 그 아래로 정렬된다.

**내보내기/가져오기**: 내 문장 화면 하단. 내보내기는 `captured-phrases` 배열을 JSON 파일로 다운로드, 가져오기는 파일 선택 → 파싱 → **기존 항목에 병합**(id 중복은 건너뜀, 덮어쓰지 않음). 브라우저 데이터 삭제 시 유실과 PC↔폰 이동을 둘 다 해결하는 수동 수단이다. 가져오기는 반드시 파싱 실패·형식 불일치를 처리해야 한다(잘못된 파일로 기존 데이터가 날아가면 안 됨).

## 기능 ②: 홈 대시보드

### 레이아웃

`body`의 `max-width`를 1000px → **1280px**로 넓히고, 홈 화면만 3열 그리드를 쓴다.

```
grid-template-columns: 0.95fr 1.15fr 1.15fr;   /* 넓은 화면 */
```

- **1열**: 빠르게 시작(기능 진입 카드 4개, 세로 배치) + 문장 담기 카드
- **2열**: 오늘 복습할 단어 (카드 최대 3개 + "+N개 더")
- **3열**: 오늘 복습할 문장 (카드 최대 3개 + "+N개 더")

**좁은 화면에서는 이 3열이 1열로 접힌다**(미디어 쿼리 또는 `auto-fit`+`minmax`). 폰에서는 지금과 같은 세로 스택이 되며, 순서는 위 DOM 순서 그대로다.

상단은 인사말(왼쪽) + 통계 타일 3개(오른쪽)를 한 줄에 배치한다. 통계는 지금처럼 큰 타일 3개를 가로로 늘어놓는 대신 우상단에 모아 공간을 아낀다.

### 홈에서 바로 채점

2·3열의 카드는 기존 `renderWordCard` / `renderSentenceCard`를 **그대로 재사용**한다. 즉 홈에서 SRS 등급 버튼과 북마크가 실제로 동작하고, 누르면 홈이 다시 렌더되어 목록이 갱신된다(기존 `vocab:refresh` / `sentence:refresh` 이벤트 흐름을 그대로 사용). 새 카드 컴포넌트를 만들지 않는다.

### 복습할 것이 없을 때

`buildTodayQueue`가 빈 배열이면 해당 열에 안내 문구를 보여준다(예: 단어 열 — "오늘 복습할 단어가 없어요. 단어장에서 새 단어를 익혀보세요."). 빈 열이 그냥 비어 있으면 고장난 것처럼 보인다.

### 타이포그래피

목업에서 확정된 크기(사용자가 여러 라운드에 걸쳐 조정):

| 요소 | 크기 |
|---|---|
| 인사말 / 서브 | 32px(700) / 16px |
| 통계 타일 숫자 / 라벨 | 34px(800) / 14px(600) |
| 섹션 제목 ("오늘 복습할 단어" 등) | 19px(700) |
| "모두 보기" 링크 | 14px(600) |
| 문장 담기: 안내문 / 입력창 / 버튼 / 대기 목록 | 17px / 18px / 17px / 18px |
| 빠르게 시작: 아이콘 / 이름 | 28px / 17px(700) |

단어·문장 카드 자체의 크기는 기존 `.word-card` / `.sentence-card` 규칙을 그대로 따른다(홈 전용으로 다시 정의하지 않는다).

## 테스트 방침

jsdom은 CSS를 적용하지 않으므로 **레이아웃은 테스트로 증명되지 않는다.** 초록불이 보장하는 것은 마크업 구조와 순수 로직뿐이며, 3열/1열 전환과 크기는 사람이 브라우저에서 봐야 한다. 이 프로젝트는 이미 CSS 누락으로 두 번 문제를 겪었으므로(`.hidden` 규칙 누락, 홈 카드가 배경과 같은 색), TS가 붙이는 클래스에 대응 CSS가 있는지는 별도로 확인한다.

회귀 테스트가 필요한 것:

- `CapturedPhrase` 저장·로드 (빈 목록, 추가, 수정, 삭제)
- 완성 판정(`japanese !== ''`)과 `toSentenceEntry` 변환
- `allSentences()` — 완성된 담은 문장만 포함하고 미완성은 제외하는지, 내장 40문장이 그대로 있는지
- `composableSentences()` — `reading`이 빈 문장을 제외하는지 (작문 연습에서 절대 못 맞히는 문장이 나오면 안 됨)
- 가져오기 — 정상 파일 병합, id 중복 건너뜀, **잘못된 JSON/형식이면 기존 데이터를 보존하고 실패**
- 홈 대시보드 — 복습 큐가 비었을 때 안내 문구, 있을 때 카드 렌더, 담은 문장이 문장 복습 카운트에 반영되는지

날짜를 다루는 `createdAt`은 기존 `src/streak.ts`와 같은 **로컬 날짜** 규칙을 따른다(`vite.config.ts`의 `TZ: 'Asia/Seoul'` 고정이 이미 적용되어 있음).
