# 비주얼 리디자인 — 설계 문서

## 배경 및 목표

Phase 1+2 구현은 끝났지만 스타일이 브라우저 기본값 수준(`src/style.css` 36줄, 테두리만 있는 회색 카드)이라 "누가 봐도 쓰고 싶은" 느낌이 전혀 없다는 피드백. 실제 배포된 사이트(`qkqhsm1.github.io/StudyLanguage`)를 열어보고 나온 요청. 언어학습 앱(듀오링고, WaniKani, Renshuu 등)을 참고해서 디자인 시스템을 새로 잡는다.

브라우저 기반 비주얼 브레인스토밍 툴로 여러 라운드의 시안을 직접 보면서 결정했다. 이 문서는 그 과정에서 확정된 내용을 정리한 것.

## 범위

- 앱 전체의 시각 디자인 시스템(색상·타이포·카드·배지·버튼 규칙) 확립
- 홈 화면 신규 추가(지금은 없음 — `main.ts`가 기본으로 바로 단어장으로 라우팅됨)
- 단어 카드, 카테고리 목록, 가나퀴즈, 가상 키보드에 새 스타일 적용
- 작문 연습에 힌트 기능 신규 추가(디자인 범위를 벗어나는 유일한 기능 추가지만, 리디자인 과정에서 나온 요청이라 같이 포함)
- **제외**: 다크 모드, 문어장/해석연습 화면의 별도 목업(카드·배지·버튼 스타일이 이미 확립됐으므로 같은 규칙을 그대로 적용하고 별도 시안 확인 없이 진행)

### 타이포그래피

- 일본어 텍스트 전체: **Noto Sans JP** (Google Fonts, `index.html`에 `<link>` 추가 — 별도 패키지 설치 없음)
- 한글/영문은 기존 시스템 폰트(`system-ui`) 유지
- 단어 카드: 한자/가나 단어 42px(weight 600) · 후리가나(ruby, `<rt>`) 22px(weight 700, 블루) · 로마자·뜻줄 16px(weight 600), 한자줄과 뜻줄 사이 간격 최소(margin-top ~1px)
- 가나퀴즈 큰 글자: 64px
- 가상 키보드: 가나 키 21px · 컨트롤 키(space/지우기 등) 17px
- 작문 연습 문제 카드: 라벨 16px(weight 600, 보조색) · 문제 문장 30px(weight 700)

### 컴포넌트 규칙

**단어 카드** (`renderWordCard`, `src/vocab/vocab-view.ts`)
- 발음 재생 버튼: 34px 원형, 연회색 배경, 블루 재생 아이콘
- 한자 단어는 `<ruby>`+`<rt>`로 후리가나 표시(지금은 별도 `.word-reading` div), 가나만 있는 단어는 후리가나 없이 그대로
- 뜻줄(로마자 · 한국어 뜻)은 한자줄 바로 아래 좁은 간격
- 배지 행: 카테고리 배지 + SRS 상태 배지(오늘 복습=주황, 며칠 남음=초록)
- 북마크: 🔖 이모지, 북마크 시 진하게·해제 시 옅게(opacity ~0.15~0.2), 카드 우상단
- SRS 등급 버튼 3개(모름/헷갈림/암기됨) 가로 3분할, 모름·헷갈림은 연회색, 암기됨만 블루 채움

**카드 리스트 레이아웃**
- `body`의 `max-width`를 640px → 1000px로 늘린다(카드 그리드가 아닌 다른 화면은 그냥 여백이 좀 더 생길 뿐, 깨지지 않음)
- 카드 리스트 컨테이너(`.card-list`, 문어장의 리스트 클래스)에 `display:grid; grid-template-columns:repeat(auto-fill, minmax(320px, 1fr)); gap:...`를 적용 — 미디어 쿼리 없이 뷰포트 폭에 따라 자동으로 1열(폰)~3열(PC)이 됨
- 문어장 카드(`renderSentenceCard`)도 동일 규칙 적용

**카테고리/스킬 목록** (`renderSkillList`, 문어장의 `renderCategoryList`도 동일)
- 지금은 `<ul><a>` 텍스트 링크뿐 → 아이콘 배지(40px 둥근 사각형, 옅은 색 배경) + 이름 + 개수 + `›` 화살표로 된 리스트 아이템으로 교체
- 아이콘은 카테고리명→이모지 매핑 테이블로 결정: `Basics=🔤, Greetings=👋, Cafe=☕, Travel=✈️`, 매핑에 없는 카테고리는 기본 아이콘 `📘`
- 매핑 안에 없는 실제 duome 스킬명이 더 있을 수 있으므로, 구현 시 `src/data/vocabulary.json`의 실제 스킬 목록을 확인해서 매핑을 최대한 채운다(전부 다 못 채워도 기본 아이콘으로 안전하게 처리됨)

**홈 화면** (신규, `src/home/home-view.ts`)
- 라우팅: `main.ts`의 기본 라우트(해시 없음 또는 `#/`)를 홈으로 변경. 기존 `#/vocab` 기본 동작은 홈으로 대체되고, 단어장은 명시적으로 `#/vocab`을 눌러야 진입(nav에 이미 링크 있음)
- 인사말 헤더("こんにちは 👋" + 서브텍스트)
- 통계 타일 3개: 단어 오늘 복습 개수(`buildTodayQueue` 재사용), 문장 오늘 복습 개수, 연속 학습일(스트릭)
  - **스트릭 계산은 신규 로직**: 지금 앱엔 학습일 기록이 없음. `localStorage`에 `{ lastDate: 'YYYY-MM-DD', streak: number }` 하나만 저장하는 순수 함수(`updateStreak(prev, today)`)를 추가하고, 홈 화면 렌더 시 한 번 호출한다. 규칙: `lastDate`가 오늘이면 `streak` 그대로 반환(이미 오늘 방문 기록됨) / `lastDate`가 어제면 `streak + 1`로 갱신 후 저장 / 그 외(기록 없음 포함, 이틀 이상 비었음)는 `streak = 1`로 리셋 후 저장. 이 부분은 순수 디자인이 아니라 작은 로직 추가이므로 구현 계획에 별도 태스크로 포함하고, 날짜 비교 3가지 케이스(오늘/어제/그 이전) 회귀 테스트를 둔다
- 기능 진입 카드 4개(2x2 그리드): 단어장 / 가나퀴즈 / 문어장 / 문장연습, 각각 옅은 그라데이션 배경 + 이모지 아이콘 + 짧은 설명

**가나퀴즈** (`renderKanaQuizView`)
- 문제 카드를 공용 카드 스타일(흰 배경, 둥근 모서리, 그림자)로 감싸기
- 큰 글자 64px 중앙 정렬
- 입력창: 둥근 테두리, 포커스 시 블루 테두리
- 제출 버튼: 풀폭 블루

**가상 키보드** (`renderKanaKeyboard`, `src/practice/keyboard.ts`)
- 지금처럼 맨 배경에 버튼만 나열하지 않고 카드(흰 배경, 둥근 모서리, 그림자)로 감싼다
- 히라가나/카타카나 토글을 알약(pill) 형태로: 선택된 쪽은 블루 채움, 안 선택된 쪽은 회색 텍스트
- 가나 키는 전부 동일한 연회색 스타일(특정 행에 색을 넣는 아이디어는 기각됨 — 근거 없이 예시로 넣었던 것)
- 컨트롤 행(space, っ, ー, 。, 지우기): 지우기만 빨간 계열로 구분, 나머지는 가나 키와 같은 톤

## 신규 기능: 작문 연습 힌트 버튼

리디자인 중 나온 요청. "코히" 같은 단어에서 장음(ー)·촉음(っ) 같은 특수 표기를 몰라서, 단어 자체는 알아도 계속 틀리는 경우를 막기 위함.

**UI** (`src/practice/compose-view.ts`)
- 입력창 옆에 "💡 힌트" 버튼 추가(주황 계열: 배경 `#fff4e5`, 테두리·글자 `#c8720a`)
- 입력창 테두리: 지금까지 입력한 내용이 정답(`reading`)의 접두사와 일치하면 블루, 어긋나면(오타) 주황으로 표시
- 힌트 클릭 시:
  1. 지금까지 정확히 일치한 부분(prefix) 길이를 기준으로 정답의 다음 한 글자를 계산
  2. "다음 글자는 OO예요" 안내 박스 표시
  3. 가상 키보드에서 해당 글자에 대응하는 키를 블루로 펄스 애니메이션 강조 — 이 매핑을 위해 `renderKanaKeyboard`가 "지금 강조해야 할 글자"를 외부에서 받을 수 있어야 함(신규 옵션 파라미터, 예: `highlightChar?: string`). 특수 글자(ー, っ)는 컨트롤 행의 해당 버튼을 강조
  4. 요음 글자(예: きゃ)처럼 한 키가 두 글자를 한번에 입력하는 경우, 힌트가 가리키는 다음 글자가 요음 조합의 앞/뒤 글자 중 하나라면 그 조합 키 전체를 강조(부분 강조는 불가능하므로)

**채점 로직** (`src/practice/compose-view.ts`)
- 한 문제에서 힌트를 한 번이라도 사용했다면, 정답을 맞혀도 SRS 등급을 `known`이 아니라 `confusing`으로 기록(완전히 암기한 게 아니므로). 오답이면 지금처럼 `unknown` 그대로
- `SrsState`/`SrsStore` 스키마 변경 없음 — 힌트 사용 여부는 문제를 푸는 동안의 로컬 컴포넌트 상태로만 추적하고, 채점 시점에 등급 계산에만 반영

## 구현 우선순위 (참고용, 실제 태스크 순서는 계획 단계에서 확정)

1. 디자인 토큰(색상·폰트) — `index.html`에 Noto Sans JP 링크, `src/style.css`에 색상/공통 카드 스타일 정의

3. 카테고리 아이콘 목록(단어장 스킬 목록 + 문어장 카테고리 목록)
4. 가나퀴즈 화면 스타일 적용
5. 가상 키보드 스타일 적용
6. 홈 화면 신규 추가 + 라우팅 변경 + 스트릭 로직
7. 작문 연습 힌트 기능

## 테스트 방침

- 순수 스타일(색상/폰트/크기) 변경은 기존 jsdom 테스트에 직접적 영향 없음(jsdom이 CSS를 적용하지 않으므로) — 단, 클래스명이 바뀌면 기존 테스트가 참조하는 셀렉터도 같이 고쳐야 함
- 신규 로직은 회귀 테스트 필수:
  - 카테고리→아이콘 매핑 함수
  - 스트릭 계산 함수(날짜 비교 로직 — 오늘/어제/그 이전 케이스)
  - 힌트의 "다음 글자 계산" 함수(순수 함수로 분리: `reading`과 `typed`를 받아 다음 글자 반환)
  - 힌트 사용 시 SRS 등급이 `known`→`confusing`으로 강등되는지
- `.hidden` CSS 누락 사고([[project-japanese-study-app]] 참고)가 재발하지 않도록, 새로 추가하는 시각 상태(예: 입력창 에러 테두리, 힌트 강조 펄스)는 클래스 토글 로직뿐 아니라 대응하는 CSS 규칙이 실제로 존재하는지 별도 확인
